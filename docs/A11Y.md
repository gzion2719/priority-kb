# A11Y.md — accessibility decisions and contrast pass

**Scope.** WCAG 2.1 §1.4.3 (Level AA) contrast pass for the Kramer-branded UI defined in `styles/kramer-brand.css`. Covers the 7 meaningful text-on-background color pairs ADR-0026 §3 enumerates, plus the alpha-composited surfaces (translucent pinks/purples over dark) that those pairs render against in practice.

Mechanical floor: `styles/contrast.test.ts` re-runs the computation at every gate and fails CI if any documented surface falls below threshold.

Out of scope:
- **WCAG 1.4.11** (non-text contrast 3:1 — borders, focus indicators, UI control boundaries). Tracked in BACKLOG as a follow-up.
- **WCAG AAA** (7:1) — not a project target.

---

## Methodology

Per WCAG 2.1 §1.4.3:

```
Linearize each sRGB channel c (0..1):
  c_lin = c <= 0.03928 ? c / 12.92 : ((c + 0.055) / 1.055) ^ 2.4

Relative luminance:
  L = 0.2126 * R_lin + 0.7152 * G_lin + 0.0722 * B_lin

Contrast ratio:
  R = (max(L1, L2) + 0.05) / (min(L1, L2) + 0.05)
```

Thresholds:
- **AA normal text:** 4.5:1
- **AA large text:** 3:1 (≥18pt or ≥14pt bold)

### Large-text exception affirmation

Every text surface migrated by M4.5/E is **normal text** under §1.4.3:

| Surface | Font size | Bold? | Threshold |
|---|---|---|---|
| `a:hover` | inherited body `12px` | no | 4.5:1 |
| `.chat-tool-chip.error` | `11px` | no | 4.5:1 |
| `.chat-banner.error` | inherited body `12px` | no | 4.5:1 |
| `.sensitivity-pill[data-tier="internal"]` | `0.75rem` ≈ 12px | no | 4.5:1 |
| `.sensitivity-pill[data-tier="restricted"]` | `0.75rem` ≈ 12px | no | 4.5:1 |
| `.btn.alert` text | `--fs-body` = 12px | no | 4.5:1 |

The 3:1 large-text exception does NOT apply to any surface in this PR.

### Alpha-composited backgrounds

For surfaces with `background: rgba(R, G, B, alpha)` over the page's solid `--kramer-dark`, the effective rendered background is computed via standard alpha-over compositing:

```
final_channel = source_channel * alpha + base_channel * (1 - alpha)
```

The composited hex is what `contrast.test.ts` asserts against — NOT the solid token. This was the load-bearing finding in M4.5/E's plan review: a `--kramer-pink-strong` candidate that passed 4.66:1 against solid `--kramer-dark` failed 4.43:1 against the actual rendered `rgba(190,0,120,0.12) over #121212` background. Picking text colors against the wrong base ships docs claiming AA compliance for surfaces that don't meet it.

`--kramer-purple` and `--kramer-pink` **stay** as the brand-identity fills for buttons (`.btn`, `.btn.alert`), chat user-message bubbles, and translucent decorative backgrounds. They fail as TEXT colors against the dark page bg; remediation below.

## Strong variants (M4.5/E)

Per ADR-0026 §3 alternative (a): introduce brighter `-strong` companion tokens for the failing pairs, leaving the brand-identity originals untouched. The exact hex picks below were chosen for headroom against the **actual rendered backgrounds** (including composited surfaces) — not just the solid dark base.

| Token | Hex | Min observed ratio (any surface) | Used at |
|---|---|---:|---|
| `--kramer-purple-strong` | `#d378ff` | **6.82:1** (sensitivity-pill internal composited bg) | `a:hover`, `.sensitivity-pill[data-tier="internal"]` text |
| `--kramer-pink-strong` | `#ff5fb0` | **6.33:1** (sensitivity-pill restricted composited bg) | `.chat-tool-chip.error`, `.chat-banner.error`, `.sensitivity-pill[data-tier="restricted"]` text |
| `--kramer-neutral-strong` | `#f5f5f5` | **5.57:1** (vs `--kramer-pink` solid) | `.btn.alert` text (`<Button variant="danger">` consumer) |

### Hex-pick rationale

- **`--kramer-purple-strong: #d378ff`** — ADR-0026 §3 row (a) showed `#a330d4` as an illustrative example; that value fails (3.51:1 vs dark) and was corrected in the same PR (see ADR-0026 §3 amendment 2026-06-10). The chosen `#d378ff` provides ≥6.82:1 across all 4 purple-text surfaces. It is noticeably brighter than the brand purple `#8200b4` (lightness ↑, slight hue shift toward violet) — accepted as the price of AA compliance per the ADR's framework.
- **`--kramer-pink-strong: #ff5fb0`** — `#e0399a` (the minimum-delta candidate) was reviewed and rejected; it fails AA against the composited pink-over-dark backgrounds (4.43-4.49:1 across the 3 translucent surfaces). `#ff5fb0` clears all 3 with ≥6.33:1 buffer.
- **`--kramer-neutral-strong: #f5f5f5`** — `#ffffff` (pure white) would clear 6.07:1 vs `--kramer-pink` but is a known eye-strain choice in dark-mode systems; `#f5f5f5` softens the rendering while still passing 5.57:1 — comfortable headroom over the 4.5 threshold for browser/display variance.

## Background-vs-text token split

Brand colors are used as both backgrounds (where their pairs were already passing) and text (where they failed). The split:

| Token | As background, paired with | As text, paired with |
|---|---|---|
| `--kramer-purple` | `--kramer-neutral` (PASS 5.92:1) | — (replaced by `--kramer-purple-strong` for text-on-dark) |
| `--kramer-pink` | `--kramer-neutral-strong` (PASS 5.57:1) | — (replaced by `--kramer-pink-strong` for text-on-dark) |
| `--kramer-mint` | `--kramer-dark` (PASS 14.88:1) | `--kramer-dark` (PASS — same pair, both directions OK) |

A future PR naively swapping a background to `-strong` would break the inverse pair — flagged here as the trap.

## Out of scope — BACKLOG

- **WCAG 1.4.11 (non-text 3:1).** Applies to: M4.5/C focus rings, M4 #1b filter-chip borders, alert/info banner borders, sensitivity-pill borders. Most are decorative or already pass by inspection (mint focus ring at 14.88:1, neutral borders at 13.77:1). A formal audit + spot fixes is queued.
- **App-side off-brand colors.** `#ff6b6b` is used as inline error-text color in admin forms (EditForm, DeleteForm, etc.). Contrast passes (6.82:1 vs dark) but the color is off-brand and inconsistent with `--kramer-pink-strong`. Cleanup queued for a brand-cohesion pass.
- **Real-world manual smoke.** This doc + contrast.test.ts cover the computational contract. A monitor-calibrated visual review at the typical operator's display is the final acceptance.
