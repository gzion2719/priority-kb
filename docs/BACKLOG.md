- **Worker — SIGTERM-mid-job mark-failed branch is unreachable via the poll loop (ADR-0019 §D6 intent gap).** In [api/worker.py](../api/worker.py) `poll_loop`, the `try/finally` around `await handler(job)` clears `state.current_job_id`/`current_worker_id` on BOTH normal return and exception, and the only call to `_handle_shutdown` is at loop exit (after the `while not shutdown_requested` re-check). Since SIGTERM only flips `shutdown_requested` (the handler does no DB work, by design), a SIGTERM arriving mid-handler lets the handler run to completion, the `finally` clears state, then loop-exit `_handle_shutdown` sees `current_job_id is None` → takes the no-in-flight branch. So `_handle_shutdown`'s mark-failed branch — the ADR-0019 §D6 "SIGTERM marks the in-flight job failed" behavior — is effectively dead code under the current wiring. `_handle_shutdown`'s contract is unit-tested directly ([api/tests/test_worker.py](../api/tests/test_worker.py)) and is correct in isolation; the gap is the poll_loop wiring that never reaches it with an in-flight job. Fix options (need design thought against §D6 intent): (a) observe `shutdown_requested` *after* the await but *before* the `finally` clear and route to mark-failed there; (b) accept that SIGTERM lets the in-flight job finish (graceful drain) and rewrite §D6 to match — arguably the safer semantics since the handler already owns its own mark_done/mark_failed. Surfaced in 2026-05-28 worker-orchestration-tests code-CR m4.

- **Cursor µs/ms round-trip fix for keyset pagination (M4 #1 + M4 #5 shared latent).** Postgres `timestamptz` is microsecond-precise; node-postgres truncates to milliseconds when materializing a JS `Date`. A keyset cursor read at ms precision then re-sent as a query parameter compares **strictly less** than the original µs-precise stored value — on `listEntriesForAdmin`'s `(updated_at, id) < cursor` DESC walk the boundary row is silently DROPPED from page 2; on `listStaleEntries`'s `(last_verified_at, id) > cursor` ASC walk the boundary row is silently RE-INCLUDED on page 2 (caught 2026-05-31 by the M4 #5 integration test's pagination-disjointness assertion; workaround = seed with explicit ms-precision ISO strings). **Production exposure today is bounded:** app write paths (`createEntry`/`updateEntry`) bind JS `Date` values which are ms-precise, so app-written rows round-trip cleanly. **Latent risk:** any raw-SQL writer (NOW() arithmetic, fixup scripts, future migrations, M2b job worker that may stamp last_verified_at via SQL) produces µs-precise rows that hit the truncation bug. **Fix options:** (a) change column declarations to `timestamp(3)` for ms-only precision (schema migration, breaks any µs-precise write); (b) pass cursor as `epoch_milliseconds` numeric instead of `Date` (loses index — comparing on a derived `EXTRACT(EPOCH FROM updated_at)*1000` can't use the existing btree); (c) bind cursor as `$N::timestamptz(3)` so Postgres truncates the stored value to ms before comparison (loses the index too if Postgres can't prove the truncation is order-preserving); (d) read the cursor's `last_verified_at` as text via a custom type-parser preserving µs, bind back unchanged. **Recommendation:** defer until production exposure surfaces. Until then, document the hazard inline in both helpers' doc-comments + pin via test (the M4 #5 pagination test catches the regression class). Surfaced 2026-05-31 in M4 #5 integration test.
